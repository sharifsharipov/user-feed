3. SOLID Printsiplari

S - Single Responsibility Principle

Har bir class faqat 1 ta sababga ko'ra o'zgaradi.

UseCase'lar - har biri 1 vazifa:

Repository'lar - har biri 1 entity:

PostRepositoryPost

O - Open/Closed Principle

Kengaytirish uchun ochiq, o'zgartirish uchun yopiq.

Resource<T> - functional API bilan kengaytirilgan:

sealed class Resource<out T> {
    data class Success<T>(val data: T)
    data class Failure(val error: AppError)

    fun <R> fold(onSuccess, onFailure): R
    fun <R> map(transform): Resource<R>
    fun <R> flatMap(transform): Resource<R>
    fun onSuccess(action): Resource<T>
    fun onFailure(action): Resource<T>
    fun getOrNull(): T?
}

AppError - yangi error turi qo'shish faqat subclass:

sealed class AppError(open val message: String) {
    data class NetworkError(...)
    data class ServerError(..., val code: Int?)
    data class DatabaseError(...)
    data class NotFoundError(...)
    data class UnknownError(...)
}

L - Liskov Substitution Principle

Barcha Repository implementation'lari interface'ni to'liq qanoatlantiradi. UserRepositoryImpl ni UserRepository interface orqali ishlatish mumkin - behavior o'zgarmaydi.

I - Interface Segregation Principle

Client faqat o'ziga kerakli method'larni ko'radi:

  • PostRepository - faqat Post operatsiyalari (3 method)
  • CommentRepository - faqat Comment operatsiyalari (1 method)
  • Oldin Comment PostRepository ichida edi - buzilgan ISP

D - Dependency Inversion Principle

Domain layer: interface UserRepository (abstract)
Data layer: class UserRepositoryImpl : UserRepository (concrete)
DI: singleOf(::UserRepositoryImpl) bind UserRepository::class

ScreenModel → UseCase → Repository interface (Domain da). Implementation'ni bilmaydi.

4. DRY / KISS Printsiplari

DRY - Don't Repeat Yourself

MuammoYechimFayl
Repository'larda takroriy try-catchsafeApiCall / safeDbCallSafeCall.kt
Har bir screen da Scaffold+Box+whenUiStateScaffold widgetUiStateScaffold.kt
PostItem va FavoriteItem bir xilPostCard widget (showBody param)PostCard.kt
TopAppBar konfiguratsiya takroriAppTopBar widgetAppTopBar.kt
MaterialTheme.typography.xxx uzunAppTextStyle.xxx / AppColor.xxxType.kt

safeApiCall - yagona error handling:

suspend fun <T> safeApiCall(
    dispatcher: CoroutineDispatcher,
    block: suspend () -> T
): Resource<T> = withContext(dispatcher) {
    try {
        Resource.Success(block())
    } catch (e: IOException) {
        Resource.Failure(AppError.NetworkError())
    } catch (e: ServerResponseException) {
        Resource.Failure(AppError.ServerError(code = e.response.status.value))
    } catch (e: Exception) {
        Resource.Failure(AppError.UnknownError(e.message))
    }
}

UiStateScaffold - screen boilerplate yo'q:

// Oldin (har bir screen da ~20 qator)
Scaffold(topBar = { ... }) { padding ->
    Box(Modifier.fillMaxSize().padding(padding)) {
        when (state) {
            is UiState.Loading -> LoadingContent()
            is UiState.Error -> ErrorContent(...)
            is UiState.Success -> { ... }
        }
    }
}

// Keyin (1 chaqiruv)
UiStateScaffold(
    state = state,
    topBar = { AppTopBar(...) },
    onRetry = screenModel::refresh
) { data -> /* faqat success content */ }

KISS - Keep It Simple, Stupid

  • UiState - faqat 3 holat: Loading, Success, Error
  • ApiService - har bir endpoint 1 qator
  • Domain model'lar - faqat kerakli field'lar, ortiqchasi yo'q
  • Har bir UseCase - 1 ta operator fun invoke()

6. Qatlamlar tafsiloti

Domain Layer

Eng ichki qatlam. Hech qanday tashqi kutubxonaga bog'liq emas (faqat Kotlin stdlib va Coroutines).

Model'lar:

data class User(val id: Long, val name: String, val username: String,
               val email: String, val phone: String, val website: String)

data class Post(val id: Long, val userId: Long, val title: String,
               val body: String, val isFavorite: Boolean = false)

data class Comment(val id: Long, val postId: Long, val name: String,
                  val email: String, val body: String)

Repository interface'lar:

interface UserRepository {
    fun getUsers(): Flow<List<User>>
    suspend fun refreshUsers(): Resource<Unit>
    suspend fun hasUsers(): Resource<Boolean>
}

Data Layer

DTO → Domain mapping (extension functions):

fun UserDto.toDomain() = User(id, name, username, email, phone, website)
fun UserEntity.toDomain() = User(id, name, username, email, phone, website)
fun PostEntity.toDomain(isFavorite: Boolean) = Post(id, userId, title, body, isFavorite)

Repository - Offline-first pattern:

override fun getPostsByUser(userId: Long): Flow<List<Post>> {
    val postsFlow = queries.getPostsByUserId(userId).asFlow().mapToList(dispatcher.io)
    val favoritesFlow = queries.getAllFavorites().asFlow().mapToList(dispatcher.io)

    return postsFlow.combine(favoritesFlow) { posts, favorites ->
        val favoriteIds = favorites.map { it.postId }.toSet()
        posts.map { entity -> entity.toDomain(isFavorite = entity.id in favoriteIds) }
    }
}

Presentation Layer

ScreenModel pattern:

class UserListScreenModel(
    private val getUsersUseCase: GetUsersUseCase,
    private val refreshUsersUseCase: RefreshUsersUseCase
) : ScreenModel {

    private val _state = MutableStateFlow<UiState<List<User>>>(UiState.Loading)
    val state: StateFlow = _state.asStateFlow()

    fun refreshUsers() {
        screenModelScope.launch {
            refreshUsersUseCase().onFailure { error ->
                if (_state.value !is UiState.Success) {
                    _state.value = UiState.Error(error.message)
                }
            }
        }
    }
}

9. State Management

2 ta sealed type:

TypeQatlamVazifasiHolatlari
Resource<T>Data / DomainOperatsiya natijasiSuccess(data), Failure(error)
UiState<T>PresentationUI holatiLoading, Success(data), Error(message)
Nima uchun ikkalasi ham kerak?
Resource - business logic uchun (AppError bilan). UiState - UI uchun (Loading holati + String message). ScreenModel Resource ni UiState ga convert qiladi. Data layer UI haqida bilmaydi.

13. Offline Support

Offline-first strategiya:
getUsers() va getPostsByUser() faqat DB dan Flow qaytaradi.
refresh() API dan yuklab DB ga yozadi.
Flow avtomatik yangilanadi. Internet yo'q bo'lsa oxirgi DB ma'lumot ko'rinadi.

Favorite real-time yangilanish:

combine(postsFlow, favoritesFlow) - FavoritePostEntity o'zgarganda postsFlow ham qayta emit qiladi. Yulduzcha real-time o'zgaradi.
